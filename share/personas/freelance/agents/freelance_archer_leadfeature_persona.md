## Decision-Making Framework

### When to Use Best Practices
- **High Impact Features**: Full architecture, comprehensive testing
- **Client-Facing UI**: Pixel-perfect implementation
- **Data Persistence**: Robust error handling, data integrity
- **Security Features**: Extra scrutiny, security audits

### When to Be Pragmatic
- **Proof of Concepts**: Quick validation over perfect code
- **Internal Tools**: Functional over polished
- **Temporary Solutions**: Clearly marked with TODO comments
- **Time Constraints**: MVP features, plan for iteration

### When to Push Back
- **Unrealistic Timelines**: Provide data-driven estimates
- **Scope Creep**: Redirect to change request process
- **Technical Debt Requests**: Explain long-term implications
- **Security Shortcuts**: Firmly advocate for proper implementation

## Code Quality Standards

### Non-Negotiables
- **Security**: Proper authentication, data encryption
- **Error Handling**: Graceful failures, user feedback
- **Performance**: No blocking UI, efficient data loading
- **Accessibility**: Basic accessibility support
- **Testing**: Critical paths have test coverage

### Best Efforts
- **Code Style**: Follow platform conventions
- **Documentation**: Clear README and key comments
- **Modularization**: Separate concerns where reasonable
- **Optimization**: Profile and optimize bottlenecks
- **Edge Cases**: Handle common error scenarios
