### Documentation Philosophy

Deanna's approach to documentation follows an empathetic, user-centered model:

**1. Understand the Audience**
- Who will read this documentation?
- What's their skill level and context?
- What problems are they trying to solve?
- What do they already know vs. need to learn?

**2. Identify the Need**
- What questions does this documentation answer?
- What pain points does it address?
- What would happen if this documentation didn't exist?
- How will success be measured?

**3. Structure for Clarity**
- Start with the "why" before the "how"
- Use progressive disclosure (simple first, complex later)
- Provide examples for every major concept
- Include clear navigation and table of contents
- Add search-friendly keywords

**4. Write for Humans**
- Use conversational but professional tone
- Avoid jargon or explain it when necessary
- Acknowledge complexity honestly
- Include "gotchas" and common mistakes
- Show empathy for the learner's journey

**5. Maintain Living Documentation**
- Update documentation when code changes
- Archive outdated documentation clearly
- Gather feedback continuously
- Measure usage and effectiveness
- Iterate based on real developer needs

**6. Make It Discoverable**
- Put documentation where developers look
- Use clear, descriptive titles
- Create multiple entry points (search, navigation, links)
- Cross-reference related topics
- Tag and categorize effectively

Sends an analytics event to the backend.

**Parameters:**

  - Max length: 50 characters

# ADR 003: State Management Refactoring

## Status
Accepted - Implementation in progress (Sprint 23)

## Context
Our current state management approach uses a mix of NotificationCenter, delegates, 
and direct property observation. This has led to:

- Difficult-to-track state changes across the app
- Race conditions in async operations
- Hard-to-test view controllers
- Unclear data flow, especially for new team members

Data identified 47 instances of state synchronization bugs over the past 6 months,
many caused by implicit state dependencies.

## Decision
We will adopt a unidirectional data flow pattern using Combine publishers for 
all significant state management.

### Key Principles
1. **Single Source of Truth**: Each piece of state has one authoritative source
2. **Unidirectional Flow**: State changes flow down, events flow up
3. **Explicit Dependencies**: State dependencies are declared explicitly
4. **Testable**: State changes are pure functions that can be tested in isolation

### Example
**Before:**
```swift
class GameViewController: UIViewController {
    var score: Int = 0 {
        didSet {
            updateUI()
            NotificationCenter.default.post(name: .scoreChanged, object: score)
        }
    }
    
    func updateScore(_ newScore: Int) {
        score = newScore
        // Implicit: triggers didSet, which updates UI and posts notification
    }
}
```

**After:**
```swift
class GameViewController: UIViewController {
    private let viewModel: GameViewModel
    private var cancellables = Set<AnyCancellable>()
    
    func setupBindings() {
        viewModel.$score
            .receive(on: DispatchQueue.main)
            .sink { [weak self] score in
                self?.updateScoreLabel(score)
            }
            .store(in: &cancellables)
    }
    
    func updateScore(_ newScore: Int) {
        viewModel.updateScore(newScore) // Explicit action
    }
}
```

## Consequences

### Positive
- **Predictability**: State changes follow explicit paths
- **Testability**: ViewModels can be tested without UI
- **Debugging**: Combine operators can log all state changes
- **Performance**: Can optimize with throttling, debouncing
- **Clarity**: New developers can trace state flow easily

### Negative
- **Learning Curve**: Team needs to learn Combine patterns
- **Migration Cost**: Refactoring existing code takes time
- **Boilerplate**: More initial setup code per screen
- **iOS 13+ Only**: Drops support for iOS 12 (acceptable per our support policy)

### Mitigation
- Data will lead training sessions on Combine patterns
- We'll migrate gradually, starting with new features
- Create template view controllers showing the pattern
- Deanna will maintain documentation with examples

## Implementation Plan
1. Week 1-2: Team training on Combine and unidirectional flow
2. Week 3-4: Create template and documentation
3. Week 5+: Migrate one module per sprint, starting with booking flow
